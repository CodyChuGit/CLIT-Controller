# Text Streaming Across The Board

Command Line Interface Traffic Controller (CLIT Controller IDE) should stream
active text everywhere the user expects to watch work progress. Streaming is not
only for the right-hand Agent Dock. It is a shared product and backend contract
for chat, task runs, queue execution, logs, terminals, approvals, and final
replay.

In this document, text streaming means progressive type-out: text deltas appear
as they are generated by a provider or received from a process. The user can read
and react to partial output before the full response, command, or run finishes.
It does not mean waiting for the whole output, saving it, then refreshing the UI.

## Product Decision

Every active run should expose text as it is produced or received, then preserve
the same events as the durable record. The user should be able to start work from
the Agent Dock, Tasks tab, or a queue action and see partial progress without
opening a separate terminal or waiting for a final response.

This applies to:

- Direct provider chat in the Agent Dock.
- Controller responses that create tasks, queue work, or request input.
- Queued Codex, Claude Code, Antigravity, and local command runs.
- Selected task detail in the Tasks tab.
- Global logs and task logs.
- Terminal drawers and the Terminals page.
- Approval, failure, retry, reroute, and cancellation states.

## User Experience

- Text appears incrementally while a run is active, using the real generated or
  received chunks rather than a synthetic animation after completion.
- Assistant responses type out progressively in the transcript as deltas arrive.
- Command output appends as stdout/stderr chunks arrive, even while the command is
  still running.
- Streaming rows stay readable: assistant text, command output, status changes,
  approvals, failures, and queue updates render as styled transcript or log rows.
- Long raw output streams into collapsed or scroll-contained regions by default
  when it would dominate the panel.
- Raw stdout, stderr, prompts, and logs remain available behind expanders or log
  links.
- A stopped, failed, or completed run leaves a clear final status event.
- Reconnecting the UI resumes from the last event cursor without duplicating text.
- If a provider CLI buffers output until the end, CLITC still streams status,
  command lifecycle, queue, and heartbeat events, then appends the final text when
  the CLI releases it.

## Backend Contract

The backend owns streaming. Frontend surfaces subscribe to backend events instead
of inventing their own polling loops.

Required behavior:

- Capture provider stdout and stderr incrementally from subprocesses.
- Emit text deltas as soon as they are received from the CLI or provider adapter;
  do not wait for process exit, full stdout capture, or final log writes.
- Redact secrets before events are persisted or sent to clients.
- Append every emitted chunk to the durable event log with a stable event ID.
- Broadcast events through `GET /api/events/stream` using SSE.
- Keep `GET /api/events?cursor=<id>` as a polling fallback.
- Attach each event to the workspace and, when available, provider, task, run,
  queue item, step, and log file.
- Preserve output ordering with per-run sequence numbers.
- Bound in-memory buffers while writing full redacted logs to disk.

Recommended event shape:

```json
{
  "id": "evt_000123",
  "type": "run.output",
  "createdAt": "2026-06-17T10:15:30Z",
  "workspacePath": "/path/to/workspace",
  "provider": "claude",
  "taskId": "task_abc",
  "runId": "run_xyz",
  "queueItemId": "queue_123",
  "step": "implement",
  "sequence": 42,
  "channel": "stdout",
  "textDelta": "edited frontend/src/App.tsx\n",
  "redacted": true,
  "truncated": false
}
```

Event types should include:

- `chat.delta`
- `chat.finished`
- `controller.delta`
- `controller.decision_received`
- `run.started`
- `run.output`
- `run.stderr`
- `run.heartbeat`
- `run.finished`
- `run.cancelled`
- `command.started`
- `command.finished`
- `queue.changed`
- `approval.required`
- `approval.granted`
- `approval.rejected`
- `task.status_changed`
- `task.summary_ready`

## Frontend Contract

The frontend should maintain one workspace-scoped event subscription and fan out
events to the Agent Dock, Tasks tab, Logs page, status bar, queue views, and
terminal views.

Required behavior:

- Use event IDs to dedupe resumed or fallback events.
- Render active text immediately in the relevant surface as a progressively
  growing transcript, log row, or terminal line.
- Keep selected task detail and Agent Dock synchronized when they are watching
  the same run.
- Coalesce rapid text chunks only for rendering performance; do not batch in a way
  that makes users wait for the full output before reviewing.
- Show connection state quietly in the status/footer when streaming is degraded.
- Fall back to polling without changing the visual model.
- Never show unredacted raw output.

Surface requirements:

- Agent Dock: stream provider replies, controller decisions, command output,
  approvals, queue changes, failures, and final status into the active provider
  transcript.
- Tasks tab: stream the selected task's active run into the task detail; after
  completion, render the same events as durable replay.
- Logs page: stream global redacted events and let users filter by task, provider,
  run, channel, and severity.
- Terminals: continue using PTY streams for interactive terminal sessions, but
  expose linked run/task context where terminal output belongs to a managed run.
- Status bar: update active run count, queue state, provider state, and streaming
  health from the same event stream.

## Non-Goals

- No direct provider API replacement for the current CLI-first model.
- No VS Code extension host, `.vsix` execution, vendor iframe, or real VS Code
  dependency.
- No browser-side secret redaction as the primary safety boundary.
- No final-only snapshots for active work.
- No fake typewriter animation that starts only after the backend already has the
  full output.
- No duplicate streaming systems per page.

## Acceptance Criteria

- Starting any provider chat or queued run creates visible streaming text before
  completion whenever the underlying process emits text.
- A user can begin reviewing partial generated text or command output while the
  provider or process is still running.
- The Agent Dock, selected Tasks tab detail, Logs page, and status/footer update
  from the same event stream.
- Task replay after completion is rebuilt from the same durable events shown
  during the live run.
- Output ordering is stable across stdout, stderr, status, approval, queue, and
  completion events.
- Refreshing the app resumes from a cursor and does not duplicate streamed text.
- Secrets are redacted before events are persisted or sent.
- Polling works when SSE is unavailable, but the target path is event-backed
  streaming.
- Existing traffic control, task files, policy, approvals, queue rules, and CLI
  execution remain authoritative.
