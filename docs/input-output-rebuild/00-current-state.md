# 00 — Current State of CLIT Controller Input / Output

Characterization of the input/output/streaming/protocol implementation as it
exists at the start of the rebuild (Phase 0). Produced by a 7-agent inventory of
the actual code (64 findings). It records what must be **preserved** (domain
behavior, security/policy invariants, SSE-first transport) and the concrete
**weaknesses** the rebuild addresses.

## The three planes are currently collapsed

The defining defect: there is no separation between **live narrative** (human
prose), **operational events** (typed app state), and **deterministic results**
(validated machine actions). Specifically:

- The controller's machine actions are **parsed out of the model's reply prose**
  via fenced `agentflow-*` / `agentflow` blocks ([chat_directives.py](../../backend/agentflow/chat_directives.py)) — the human-input surface carries no structured intent at all.
- The live event contract is an open `{type: str, data: dict}` ([event_bus.py](../../backend/agentflow/event_bus.py)); the frontend re-derives meaning from `type` strings and `data` shapes ([displayModel.ts](../../frontend/src/lib/displayModel.ts), [stream.tsx](../../frontend/src/stream.tsx)).
- Input destination/intent are **ambient UI state** (active channel tab + engine
  dropdown), not part of the submission.

The rebuild introduces the three planes as validated, versioned contracts:
[io_contracts.py](../../backend/agentflow/io_contracts.py) (input + operational
events), [controller_protocol.py](../../backend/agentflow/controller_protocol.py)
(deterministic controller result), and [contracts.py](../../backend/agentflow/contracts.py)
(summaries). See [01-architecture.md](01-architecture.md) and
[02-protocols.md](02-protocols.md).

## Findings by area

## Input surfaces (frontend composers, send paths, API layer, drafts, keyboard/busy behavior)

There is ONE shared input component, Composer.tsx, rendered at three places: the Agent Dock (ChatPanel.tsx), the Tasks page "Continue task" card (TasksPage.tsx), and — wait, only two. StepChat.tsx is display-only (a "Run" button, no text box). Composer is a single uncontrolled-style <textarea> with a send/stop button. Intent and destination are NOT encoded in the input at all: where a message goes is decided entirely by ambient state outside the composer — the active channel tab (ChatPanel's `channel` state, ORCH vs provider id) selects chatSend vs chatDirect; the engine-picker dropdown sets `provider`. The Tasks "Continue" composer hardcodes chatSend(msg, null) regardless of which task is open — the task id shown in the chip is decorative, never sent to the backend, so "continue THIS task" actually just messages the global controller. Drafts are never persisted: composer text lives in transient useState (`input`, `continueText`) and is lost on reload, workspace switch, or tab switch (no persist.ts usage for any input). persist.ts only stores UI chrome (open tabs, panel widths, last selected task). Keyboard behavior (Enter=send, Shift+Enter=newline) is duplicated inconsistently across surfaces: Composer uses Enter-to-send; commit message uses Cmd/Ctrl+Enter; workspace path and preview URL use plain Enter; CommandPalette uses Enter-to-run-top-match. Busy/disabled logic is computed differently in each host. Backend accepts message (1–20000 chars) + optional provider; server re-trims an already-trimmed string; provider validated against AGENT_PROVIDER_IDS. Controller "intent" (create task / queue / run command) is not expressed by the user's input — it is parsed out of the model's REPLY via fenced agentflow blocks (chat_directives.py), so the human input surface carries no structured intent whatsoever.

- **Composer is the single shared text-input primitive; one <textarea>, no intent/destination encoding** — Destination/intent live outside the input as ambient UI state (active tab, dropdown), so the same keystrokes mean different things depending on hidden context. A rebuild should make the input itself carry an explicit, typed {destination, intent} so the action is self-describing and testable rather than reconstructed from sibling state.
- **Destination is selected by the channel tab + engine dropdown, branched at send-time** — Two endpoints (chatSend/chatDirect) with divergent meaning behind one box and one Send button. The branch is implicit; there is no preview of what mode you're in beyond the tab highlight. Consolidating onto one typed submit contract (message + target descriptor) would remove the at-send branching and the duplicated busy/stop wiring.
- **Tasks 'Continue task' composer ignores the task it claims to continue** — The UI promises task-scoped continuation but delivers an unscoped controller message; the chip is misleading and the response surfaces in a different panel. A rebuild needs a real task-scoped input contract (carry taskId to the backend) and in-place reply rendering, or the affordance should be removed.
- **No draft persistence anywhere — input text is ephemeral useState** — Any half-typed prompt is lost on reload/workspace-switch with no recovery. The rebuild should persist per-(workspace,channel) drafts through persist.ts (or similar) and restore them, matching how tab/width are already persisted.
- **Keyboard submit semantics are inconsistent across every input** — No single source of truth for submit/keyboard behavior; users learn one Enter convention then hit a different one in the next box. A rebuild should centralize key handling (or at least document one rule) so Enter/Shift+Enter/Cmd+Enter behave predictably everywhere.
- **Composer textarea auto-grow and disabled/busy states are recomputed by hand per host** — Send-guard, busy-tracking, and trim logic are duplicated in every host with subtle differences (TasksPage has no disabled-while-busy on the textarea; busy semantics differ — `pending` vs local `continuing`). The auto-grow heuristic mis-measures wrapped lines. Consolidate the in-flight guard, trim, and sizing into the Composer itself so hosts only supply an async onSubmit.
- **Backend send contract is thin and re-trims; user input carries no structured intent** — The input->backend contract is just free text + optional provider; there is no typed user intent, no idempotency/dedupe key, and a redundant double-trim. Status handling on the frontend is stringly-typed (status === 'started'/'provider_missing' magic strings) and only partly handled. A typed I/O contract (Phase 1) should replace the string-status switch and let the user (not just the model) express structured intent if desired.

## Live output + streaming (SSE/polling, accumulation store, delta emission, render)

End-to-end streaming path: a subprocess's stdout/stderr is read in 4KB chunks by process_runner._read_stream, redacted and whitespace-split into "deltas" (run.output / run.stderr / chat.delta / controller.delta), and published to an in-process EventBus ring buffer with a process-monotonic integer `id`. The frontend keeps ONE SSE connection (GET /api/events/stream) with a 1.5s polling fallback (GET /api/events), both resumed by `cursor` (= last `id`). The StreamStore (stream.tsx) dedupes strictly by `id > cursor`, accumulates per-runId text into RunStream.stdout/stderr (string concat, capped at 300KB tail), and coalesces React notifications to one rAF flush. Consumers (ChatPanel, StepChat, TaskViews.LiveOutput) read the accumulated string via useRunStream(runId) and pass it to SmoothStreamingText, which reveals newly-appended chars over animation frames. Structural lifecycle events (run.started/finished, queue.*, task.*, approval.*) flow through state_store.append_event which persists to events.json AND mirrors to the bus; high-frequency text deltas are NOT persisted (full output goes to a per-run log file on disk). The design is largely sound but has several concrete duplication/reanimation and resume-correctness weaknesses driven by (1) two independent id sequences, (2) an in-memory-only bus with no replay of deltas, and (3) the live-string→persisted-message handoff ordering.

- **Backend delta emission: chunk read, whitespace-safe split, redact** — Deltas are redacted twice (carry-split in runner emits whole tokens, then redact() again in publish) — wasted CPU per chunk on hot path; minor. Bigger: the carry/whitespace-hold means a fast burst of single tokens with no whitespace can be withheld until 64KB accumulates, so 'live' output can stall visibly for whitespace-less streams. Delta boundaries are byte-chunk-driven (4096) not semantic, so multibyte UTF-8 at a chunk edge is decoded with errors='replace' per chunk and can corrupt characters spanning a 4KB boundary.
- **EventBus: single in-memory ring buffer, monotonic id, NO delta durability** — In-memory only + bounded at 4000 events. During a high-volume run, deltas churn the ring faster than a paused/hidden tab polls (POLL_MS=1500, and polling is skipped while document.hidden), so events older than the buffer window are silently dropped — the cursor jumps forward and the UI shows a GAP in live text with no indication. On backend restart _seq resets to 0 while the frontend cursor is large, so apply()'s `id <= cursor` guard (stream.tsx:81) DISCARDS every event from the restarted backend until _seq climbs back past the stale cursor — live output appears frozen after a restart until a workspace switch resets the store. The ws==None match means events with null workspacePath leak across workspaces.
- **TWO independent id sequences (durable events.json vs in-memory bus)** — A structural event has different ids in events.json vs the live stream, so the two cannot be reconciled/deduped against each other. There is no replay of durable events into the bus on restart, so the bus-backed UI cannot recover missed structural transitions after a restart even though they exist on disk — the durable ledger is effectively write-only for the live UI path.
- **SSE endpoint: polling loop inside a generator, 0.25s tick, Last-Event-ID honored** — The SSE 'stream' is really server-side polling of the in-memory buffer every 250ms — adds up to 250ms latency per delta and burns a worker per open connection doing busy-wait. limit=1000 per tick with 0.25s sleep caps throughput at ~4000 events/sec; a faster-emitting run will lag and can fall behind the 4000-event ring (see EventBus finding). No max-connection guard; every browser tab opens its own StreamingResponse generator.
- **Frontend StreamStore: id-dedup, per-runId string accumulation, rAF coalescing** — Dedup is purely by global `id`; if the backend ever restarts mid-session (id resets) all new events are dropped (see finding 2). cap() slices the last 300KB on EVERY delta append — for a long run this re-slices a 300KB string per chunk (O(n) per delta, O(n^2) total). The store never garbage-collects finished RunStreams (only the 250-event `recent` array is bounded; the `runs` Map grows unbounded across a session). updatedAt is a synthetic counter, not a timestamp, so consumers can't order/expire by real time.
- **SSE/polling connection manager (EventStreamProvider) and reconnect** — On transient SSE drop it shows 'polling' but does NOT actually start the polling timer — it relies solely on the browser's native EventSource reconnect; if the browser's reconnect also stalls, no fetch fallback runs and live output silently halts while the badge merely reads 'polling'. Polling is fully paused while the tab is hidden, so a backgrounded tab returning after a long run resumes from a cursor whose deltas have already churned out of the 4000-ring → permanent gap. The fetch() calls use no AbortController/timeout.
- **SmoothStreamingText reanimation guards (presentation-only)** — The 'never animate backwards' guard keys on text.startsWith(prev). When the 300KB cap() in the store finally trims the head, the new text no longer startsWith the old prefix, so the whole (capped) string reveals instantly — a visible flash/jump on very long runs. Also, because revealed state lives in refs keyed to component identity, if a consumer remounts the component (e.g. ChatPanel switches the pending block to a persisted Message on finish) the text re-snaps; combined with finding 8 this is the core reanimation surface.
- **Live→final handoff ordering risk (chat reanimation/duplication)** — chat.finished is emitted BEFORE on_complete persists the message and clears _pending. A load() triggered by that event can race: it may fetch state where _pending is still set (message not yet appended) → the streamed reply keeps showing as 'pending/thinking'; a subsequent poll then shows the final message. There is a window where both the live RunStream stdout (still in the store, never cleared) AND the persisted message exist, and the UI chooses based on whether `pending` is null — a flicker/double-render of the same reply. The RunStream for a finished chat is never evicted, so its full stdout lingers in memory and could re-surface if pending.runId is ever reused or re-read.
- **Structural-event refetch fan-out (load storm mitigation already in place)** — Every structural event (queue.*, task.*, run.started/finished, approval.*) bumps a single global structuralRev, so during active orchestration each surface re-runs its full load() (multiple API calls) on every transition — a coarse invalidation with no per-entity granularity. The code comments document a prior fetch-storm bug from including structuralRev in the wrong effect deps; the underlying coarse-grained invalidation remains a scalability weakness for the rebuild.

## Event contract (backend emission + frontend interpretation)

There are TWO parallel, untyped event streams. (1) The LIVE event-bus stream (event_bus.py BUS.publish) carries a flat dict keyed by a free-form `type` string and a generic `data: dict`; it powers SSE (/api/events/stream) and polling (/api/events). (2) The DURABLE task.json log (task_service._add_event) is a SEPARATE event vocabulary read directly from task metadata. Most structural transitions are emitted to BOTH via state_store.append_event (which persists to events.json AND mirrors to BUS), but with DIFFERENT `type` strings (e.g. durable `task.created` vs task.json `task_created`). Nothing is schema-validated on either side: `data` is an opaque dict, the frontend re-declares the shape by hand (streamEvent.ts coerce, displayModel.ts string maps), and `data` keys are read positionally by string (`data?.status`, `data.runId`). The biggest domain-logic-in-strings problem is NOT the event maps (those at least key off the type enum) but taskFormat.ts, which re-parses raw agent prose with regexes (errors/warnings/changed-files/test-tallies/command-titles) to manufacture UI state the backend never emitted structurally — exactly the freeform-prose-parsing the displayModel header warns against. Detail strings are human prose assembled with f-strings on the backend and rendered verbatim as the only card "summary" bullet; structured `data` is largely emitted but barely consumed.

- **Two event vocabularies: live bus type strings vs task.json type strings (no shared enum)** — Two+ parallel untyped event vocabularies with overlapping-but-divergent names and three different on-wire shapes; no single source of truth, no enum, no validation. A rebuild needs one typed event taxonomy emitted once.
- **`data` is an opaque dict whose shape is per-type, implicit, and inconsistent** — `data` is a per-call ad-hoc bag with inconsistent keys for the same logical field (status, runId, reason) across event types and even across branches of the SAME type — impossible to consume safely. Rebuild needs a typed, per-event payload.
- **Frontend re-declares the wire shape by hand and reads `data` by stringly-typed key lookup** — The contract is duplicated in TS by hand and `data` is read with untyped key lookups + magic-string comparisons ('succeeded'); a shared typed schema (codegen or hand-shared) would eliminate the parallel re-declaration and stringly-typed reads.
- **Two big event→UI string maps decide card type/severity/style from the type word** — Card semantics are derived from the type word via lookup maps (acceptable) but DUPLICATED across two vocabularies and re-augmented with magic-string status checks; consolidating to one event taxonomy collapses both maps into one.
- **taskFormat.ts manufactures UI state by REGEX-PARSING raw agent prose (the core anti-pattern)** — Errors, warnings, changed-files, test results, command titles, and budget health are RECONSTRUCTED in the browser by regex over raw prose/log text because the backend never emits them as structured event fields. This is the highest-value thing the rebuild must move to typed emission.
- **`detail` is backend-assembled human prose used directly as the card summary; structured fields under-emitted/under-consumed** — Human-readable content is locked into backend prose f-strings (not localizable, not re-renderable), while genuinely structured fields (failureKind, exitCode, durationMs, from/to) are emitted but dropped by the live projector. Rebuild should carry a typed HumanSummary + structured fields and render from them, not from a prose blob.
- **Live deltas are unvalidated, unpersisted, and routed purely by type-string + channel** — The live-vs-durable and stdout-vs-stderr and chat-vs-controller-vs-run distinctions are all encoded in type-string suffixes/prefixes + a channel field + stream_kind, with no typed delta schema; reconnection/dedupe relies solely on a numeric id and these conventions.
- **Transport: SSE and polling ship the raw bus dict as JSON with no event-name framing** — Because no SSE `event:` name is used, the discriminator is buried in the JSON `type` string — every consumer must string-switch on it; a typed/multiplexed transport (named SSE events or a discriminated-union frame) would let consumers dispatch without re-parsing the type word.

## Controller directives & deterministic output (chat_directives.py, contracts.py, prompt_templates.py, chat_service.py)

The controller emits decisions in TWO parser-supported formats: (1) a native fenced `agentflow` JSON envelope `{"version":"1","decisions":[{"kind":"task|queue|run|done|needs_user", ...}]}` (bare list or single object also accepted), each item validated against versioned Pydantic contracts and silently dropped if invalid; and (2) legacy markdown `agentflow-task`/`-queue`/`-run`/`-done`/`-needs-user` fenced `key: value` blocks. Every `parse_*` reader is "structured-first with markdown fallback" (chat_directives.py:78-187): if ANY valid structured decision exists in the reply, ONLY the structured path is used and ALL legacy blocks are ignored; otherwise the markdown blocks are parsed. Parsing is per-kind and independent — there is no single "parse this reply into one decision set" entrypoint, so each consumer re-scans the full text 4-5 times (each `parse_*` re-runs `extract_structured_decisions`, which re-regex-scans + re-JSON-parses + re-validates). Validation lives in contracts.validate() and fails safe (unknown kind / wrong version / schema error → structured FailureRecord, never raised). Execution happens in two places in chat_service.py: the interactive chat loop `send.on_complete` (lines 360-444) handles task/queue/run; the system-initiated `orchestrator_consult.on_complete` (lines 612-728) handles queue/done/needs_user/run. Commands from `run` directives are gated by policy_service.classify_action (allow / require_approval / deny). Machine actions are NOT inferred from prose anywhere — only fenced blocks drive actions; prose is stripped via strip_action_blocks and shown as reasoning. Key gaps for the rebuild: the typed contracts exist but are essentially dead in production (only consumed in a test); execution semantics differ between the two consumers; the native JSON block renders as raw JSON to the user in the frontend; and there is no enforced 1:1 between a decision and its execution (each kind parsed independently, task limited to first, run capped at 3, queue/done/needs_user to first match).

- **Two directive formats, structured-first with full markdown fallback** — Structured-first/markdown-fallback is decided per-reader by 'did extract_structured_decisions return non-empty', not per-block. Consequence: if the reply contains one valid structured decision of ANY kind, every other kind's legacy markdown block is silently ignored — a mixed reply (structured run + legacy task) loses the task with no signal. There is no single 'parse the reply once into a decision list' API; each kind is parsed independently from scratch.
- **extract_structured_decisions is re-run per kind — redundant scan/parse/validate** — Validation and JSON parsing run 3-4 times per reply. The rebuild should parse+validate the reply into a single typed decision list once, then dispatch. Also: validation FailureRecords produced inside extract_structured_decisions are discarded (line 62 ignores _failure), so the controller/user never learns WHY a malformed decision was dropped — silent failure.
- **Versioned typed contracts exist but are dead in production** — The 'deterministic semantic layer' (Pillar 5) is half-wired: directive contracts validate input but the validated typed objects are immediately .model_dump()'d back to dicts and consumed as loose dicts; the richer result/summary contracts (test results, command results, task summaries, hand-offs) are never produced or emitted anywhere — the UI gets prose+events, not typed records. The rebuild should make these the actual on-the-wire shapes, not aspirational classes.
- **Execution semantics diverge between the two directive consumers** — Two separate hand-written dispatchers with different rules: chat allows task+queue+run together but no done/needs_user; consult allows run + exactly one of queue/done/needs_user. A 'queue then done' or 'done with a final run' combination behaves differently depending on which path the reply arrives through. No unified executor; ordering and mutual-exclusivity are ad hoc. The rebuild needs one deterministic dispatch table keyed by decision kind.
- **Run-command directives: policy gate, caps, and the approval/manual-mode side doors** — The run path is the only place a directive triggers real shell side effects, and its policy is solid, but the approval result is emitted as prose system messages + loose task events, not as the typed ApprovalRequest contract that already exists in contracts.py. The cap of 3 is enforced at parse time, silently truncating extra run decisions with no feedback.
- **Prompt templates and parsers can drift; orchestrator prompts hand-author both formats** — The directive grammar is duplicated in three hand-maintained places — the prompt strings, the regex/key parsers in chat_directives.py, and the Pydantic contracts — with nothing generating one from another. Step ids, kinds, and the version='1' literal must be kept in sync by hand. The 'gemini_qa' id vs 'antigravity' provider naming mismatch is exactly the kind of drift this invites. The rebuild should derive the prompt grammar and the validator from one schema.
- **Native agentflow JSON block renders as raw JSON in the chat UI** — The PREFERRED output format the prompts push the controller toward (native agentflow JSON) is the one the UI cannot render — the user sees a raw JSON blob instead of an action card, while the deprecated legacy format gets the nice colored card. The rebuild must render whichever format is canonical, ideally from the same typed decision the backend executed rather than re-parsing message text in the browser.
- **displayModel is structured-from-events, but controller decisions aren't part of that taxonomy** — The clean event→card projection is the right model, but the controller's actual decisions reach this layer only as side-effect events (queue.enqueued, task.summary_ready, controller.decision_received with just {runId}) emitted ad hoc by chat_service after executing a directive — the decision itself is never an event carrying its typed contract. So 'controller decided X' is reconstructed from downstream effects, and the chat-channel directives (which never become task/stream events) bypass this layer entirely and rely on Markdown.tsx prose rendering. The rebuild should emit the validated decision as a first-class typed event that displayModel projects directly.

## Output presentation (frontend rendering of prose/code/commands/tools/approvals/errors/summaries/logs)

The frontend has TWO parallel, disconnected rendering pipelines for agent output. (1) A "prose" pipeline: a hand-rolled `Markdown` component renders agent replies (chat bubbles + task step replies) with bold/code/headings/lists/tables/fenced-code + custom "directive" cards. (2) A "structured/deterministic" pipeline: `displayModel.ts` projects backend task/stream events into a fixed `CardType` taxonomy rendered by `TimelineCard`, with deterministic regex summarizers (`taskFormat.ts`) for logs/commands and `RawDetail` for paginated raw text. ANSI is stripped only in the prose `RawDetail` path. The two pipelines never meet: the controller dock (`ChatPanel`) and `StepChat` re-implement their own message/bubble lists inline rather than using the supposedly-canonical `ConversationView`/`Message`, which has tests but ZERO production consumers. The markdown renderer is a custom regex/split parser (no library), which is XSS-safe (no dangerouslySetInnerHTML) but only supports a thin subset of markdown and silently mis-renders anything outside it. Same agent prose renders richly in chat but as a single un-formatted bullet line in TimelineCard summaries. Error/warning/changed-file detection is purely heuristic regex over stdout (false positives like "no errors" → error). Visual hierarchy is built almost entirely from tiny text sizes (text-[10px]/[11px]) and a sprawling, duplicated color-by-role/status/severity/health system defined in ~8 separate maps across files.

- **Custom hand-rolled markdown parser — no library, thin subset, silent failure modes** — A bespoke regex parser handles only ~6 block types and 2 inline types; anything richer (links, nested formatting, code in fences without trailing newline, indented tables) is dropped or shown as raw text. For a rebuild, replace with a real markdown lib (remark/markdown-it) + a sanitizer, or define a strict structured output contract so the UI never parses freeform prose.
- **Two disconnected pipelines: freeform-prose Markdown vs deterministic CardModel — and they disagree on the same data** — Two renderers for the same semantic content produce inconsistent output; the 'deterministic' card path drops all inline formatting and caps to ≤5 bullets, while the prose path keeps everything. A rebuild should unify on one model: structured cards whose text fields render through the same inline renderer.
- **ConversationView/Message claim to be the 'single canonical chat renderer' but have ZERO production consumers** — The 'canonical' component is dead code; ChatPanel and StepChat each reimplement message lists, auto-scroll, live-streaming panes, and bubble chrome. A rebuild should actually route every surface through one ConversationView, or delete it and own the divergence honestly.
- **Card taxonomy is large, duplicated across two event maps, and partly aspirational** — Two parallel event→card maps with copy-pasted demotion logic and several unreachable card types. A rebuild should have ONE event source of truth (the backend emits the card type) instead of the frontend re-deriving it twice.
- **Errors/warnings/changed-files/test-tallies are heuristic regex over raw stdout — lossy and false-positive-prone** — All 'what happened' signals are reverse-engineered from unstructured CLI text with broad regexes, guaranteeing both misses and false positives. A rebuild should get errors/warnings/changed-files/test-results as structured fields from the agent/runner, not scrape them from stdout.
- **ANSI handling is partial and split across modes — stripped in some prose views, left raw in others** — ANSI is inconsistently handled: stripped in one component, rendered as literal escape bytes everywhere else, and never shown as actual color. summarizers run on un-stripped text. A rebuild needs one ANSI policy (strip-to-plain or render-to-color) applied uniformly to every log/stdout view.
- **Commands & approvals: command-as-card with $-prefixed mono line; approvals are heuristic-styled action cards** — At least three different ways to present a command ($-mono line) exist (CommandCard, DirectiveCard 'agentflow-run', SystemNotice '$' detection), each with its own styling and detection heuristic. Approval 'kind' and system-notice category are both inferred from string prefixes/regex rather than typed fields.
- **System notices and 'directives' are parsed from string prefixes / fenced-lang — fragile classification** — Two more classifiers driven by string prefixes (notice category) and fenced-code language sniffing (directive type) with key:value body parsing — brittle and invisible to type-checking. A rebuild should make these typed message kinds from the backend, not prose conventions the UI reverse-engineers.
- **Visual hierarchy is carried by micro-typography and a sprawling, duplicated color system** — Tiny near-uniform font sizes give weak scannability, and the same concepts (step hue, provider color, status color, health color) are re-encoded in many uncoordinated maps that already disagree (blocked amber vs rose). A rebuild needs ONE shared design-token source for status/severity/role/health color + a real type scale.
- **RawDetail is the one solid, shared primitive — paginated read-only viewer for all machine text** — none — keep. This is the well-factored, genuinely-shared raw-text viewer; preserve it (and route ALL raw output through it) in the rebuild. Only gap: ANSI stripping is limited to 3 kinds here and absent in the live/streaming <pre> paths elsewhere.
- **Live streaming output: SmoothStreamingText is shared, but its live <pre> wrappers are duplicated per surface** — The animation primitive is shared, but the live-output container (height, char cap, ANSI, styling) is reimplemented 3 times with different magic numbers. Consolidate into one LiveOutput used everywhere, with consistent ANSI handling and caps.

## Frontend cross-surface I/O: message rendering, auto-scroll, replay-vs-live, accessibility

The four output surfaces (Agent Dock / ChatPanel, Tasks page + pages/tasks/*, Logs page, and the live event-driven cards) each interpret and render agent I/O independently, despite a "Pillar 4 consolidation" that produced shared primitives (ConversationView, useAutoScroll, cardFromStreamEvent) that are NEVER imported by any surface. ChatPanel.tsx is a 859-line monolith owning persistence, multi-channel polling, busy-state computation, unread tracking, command-palette wiring, and three competing inline renderers (system notices via Message, live agent stdout via raw <pre>+SmoothStreamingText, agent activity via LiveOutput). Auto-scroll is implemented four times: one shared hook (useAutoScroll, with the only "scroll-up-to-read" + "New output" affordance) that nothing uses, and three hand-rolled `el.scrollTop = el.scrollHeight` effects (ChatPanel, StepChat, TaskViews.LiveOutput) that yank the user to the bottom on every update with no escape. Replay (StepChat, reconstructed Exchange pairs) and live (ChatPanel pending + useRunStream deltas) diverge structurally: replay renders agent output through Markdown, live renders the same provider's bytes as raw monospace; replay has no provider duration/attribution, live does; the canonical Message renderer is used only by ChatPanel and the dead ConversationView, never by StepChat. ANSI stripping only happens inside RawDetail, so every live/inline pane (ChatPanel, StepChat, LogConsole, LiveOutput) shows raw escape codes. Accessibility: streaming output regions are not announced (only LogConsole's running block has aria-live; ConversationView's scroll region and all live panes lack role="log"/aria-live), and the "New output" affordance only exists on the unused component. Eight independent setInterval pollers run alongside the shared SSE stream.

- **ChatPanel.tsx is an 859-line monolith mixing polling, parsing, state, and three renderers** — Single component conflates transport (polling), data selection, presentation, and three renderers; no surface can reuse its conversation logic. Rebuild should split into a transport/store layer, a channel-state selector, and a presentational transcript that delegates to ONE message renderer.
- **The 'shared' consolidation layer (ConversationView, useAutoScroll, cardFromStreamEvent) is dead code — zero importers** — The intended single-source-of-truth renderer/scroll/projection exists but is unwired; real surfaces re-implement all three. Rebuild must either adopt these and delete the inline duplicates, or delete the dead layer — keeping both is the core inconsistency risk.
- **Auto-scroll is implemented 4 times with divergent behavior; 3 give the user no way to read scrollback** — During active streaming the user cannot scroll up to read earlier output in the dock or step panes — every delta yanks them back down. Rebuild needs ONE scroll behavior (the useAutoScroll semantics) applied to every output pane.
- **Four competing message/output renderers; the 'canonical' Message is used by only 1 live surface** — Same agent reply renders differently in dock vs task replay (bubble shape, attribution, duration, collapse threshold). Rebuild needs one message component consumed by every surface, parameterized by density — not re-implemented per page.
- **Replay vs live diverge in both data model and rendering for the same agent output** — Streaming output (mono, no markdown) visually 'reflows' into markdown on reload, and timing/attribution appears/disappears. Two separate live-overlay mechanisms (ChatPanel pending vs StepChat isLive). Rebuild needs one I/O record that is identical live and replayed, rendered by one component.
- **ANSI escapes are only stripped inside RawDetail; every live/inline pane shows raw escape codes** — CLIs that emit color codes (most do) show literal escape sequences in the dock, step panes, and logs — only the paginated RawDetail expander is clean. Rebuild should normalize/render ANSI once at the output-rendering boundary, not only in one viewer.
- **Streaming output regions are not announced to assistive tech; affordances missing** — Screen-reader users get no announcement of streaming replies/output in the primary surfaces, no keyboard nav for the engine picker, and no resume-scroll control. Rebuild should give every transcript a role='log'/aria-live region and a consistent keyboard-accessible scroll affordance.
- **Logs page splits the same event stream into two unrelated renderers; live cards projection unused** — Live events render with a different taxonomy/colors than the card surfaces, and the structured event->card projection that would unify them is dead. Rebuild should drive Logs from the same displayModel/TimelineCard projection used elsewhere and consolidate the color maps.
- **Eight independent setInterval pollers run beside the shared SSE stream, with ad-hoc refetch timing** — Redundant transport: SSE deltas + structural-revision refetch + N independent intervals all fetch overlapping snapshots, with documented fetch-storm fragility. Rebuild should make the shared event store the single source for refresh, with polling only as the explicit fallback the store already manages.
- **Markdown/directive parsing is regex-based, stateful, and re-runs per render on streaming text** — Custom regex markdown mis-renders mid-stream and is duplicated/forked from the raw renderer; step metadata is defined in four places. Rebuild should use a real incremental markdown parser (or render-as-mono-until-complete consistently) and a single step taxonomy module.
